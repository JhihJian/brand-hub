# 品牌中台（Brand Hub）产品需求文档

> 版本：v1.1｜状态：Draft｜最后更新：2026-02-26

---

## 1. 文档概述

### 1.1 背景

我们正在构建一套"品牌中台 + 轻量子产品"的生态体系。品牌中台（Brand Hub）作为整个生态的基础设施层，统一管理用户身份、会员资格和访问控制。各子产品（如饮食管理、健身追踪等）作为纯业务单元接入，不持有任何用户管理能力，仅通过品牌中台获取用户身份与会员状态。

### 1.2 产品定位

品牌中台是一个**内部基础设施服务**，不直接面向终端用户呈现独立界面。它通过 API 为所有子产品提供统一的认证、授权和会员管理能力。终端用户感知到的是一个统一的品牌账号体系——一次注册，多产品通用。

### 1.3 核心原则

- **品牌统一**：用户只有一个品牌账号，所有子产品共享该身份。
- **邀请准入**：非开放注册，通过邀请码控制用户增长节奏，邀请码可预设会员等级。
- **会员集中**：会员资格由品牌层统一管理，子产品只读 Token 中的会员状态。
- **子产品无状态**：子产品不存储用户信息，只持有用户标识符（`user_sub`）和昵称（`nickname`，来源于 Token）用于关联业务数据。
- **极简轻量**：拒绝重型方案，追求低资源占用和高可维护性。

## 2. 用户角色

| 角色 | 说明 | 权限范围 |
|------|------|----------|
| **访客** | 未注册用户 | 无任何权限，需持有邀请码方可注册 |
| **普通用户（Free）** | 已注册但未订阅会员 | 可使用子产品的免费功能 |
| **会员用户** | 持有月度/年度/终身会员 | 可使用子产品的全部功能（具体权限由子产品定义） |
| **品牌管理员** | 运营人员 | 管理邀请码、查看用户、管理会员状态 |

## 3. 功能需求

### 3.1 用户注册与登录

#### 3.1.1 邀请码注册

**用户故事**：作为一个持有邀请码的访客，我希望通过手机号 + 短信验证码 + 邀请码完成注册，以获得品牌账号。

**业务规则**：

- 注册必须提供有效的邀请码，无邀请码不可注册。
- 每个邀请码仅可使用一次，使用后标记为已消耗。
- 邀请码可设置过期时间，过期后不可使用。
- 邀请码可预设会员等级（free / monthly / yearly / lifetime），注册成功后用户自动获得该等级的会员资格。若邀请码未指定等级，默认为 free。
- 手机号全局唯一，不可重复注册。
- 注册时需设置昵称（nickname），后续可修改。

**注册流程**：

```
访客输入手机号 + 邀请码
  → 校验邀请码有效性（存在、未使用、未过期）
  → 发送短信验证码
  → 访客输入验证码 + 昵称
  → 校验验证码
  → 校验手机号唯一性
  → 创建品牌账号
  → 根据邀请码预设等级创建会员资格
  → 标记邀请码为已使用
  → 签发登录凭证
  → 注册完成，进入产品
```

#### 3.1.2 登录

**用户故事**：作为已注册用户，我希望通过手机号和短信验证码登录，获得访问所有子产品的凭证。

**业务规则**：

- 仅支持手机号 + 短信验证码登录，无密码。
- 短信验证码有效期 5 分钟，每个手机号 60 秒内不可重复发送。
- 登录成功后签发短期访问凭证和长期刷新凭证。
- 未注册手机号尝试登录时，返回明确错误提示（非自动注册）。
- 支持多设备同时登录，每个设备独立管理会话。

**登录流程**：

```
用户输入手机号
  → 校验手机号已注册
  → 发送短信验证码
  → 用户输入验证码
  → 校验验证码
  → 签发 Access Token + Refresh Token
  → 登录完成
```

#### 3.1.3 登出

**用户故事**：作为已登录用户，我希望可以安全退出当前设备的登录状态。

**业务规则**：

- 登出仅影响当前设备/会话。
- 登出后刷新凭证即时失效。

#### 3.1.4 凭证刷新

**业务规则**：

- 访问凭证有效期短（分钟级别），到期后通过刷新凭证获取新的访问凭证。
- 刷新凭证有效期较长（天级别），到期后需重新登录。
- 每次刷新时，凭证中的会员状态和用户昵称自动同步为最新值。

---

### 3.2 短信验证码

#### 3.2.1 验证码发送

**业务规则**：

- 验证码为 6 位纯数字。
- 同一手机号 60 秒内仅可发送一次（防刷）。
- 同一手机号每日发送上限 10 条。
- 同一 IP 每小时发送上限 20 条。
- 验证码有效期 5 分钟，验证成功后立即失效。
- 验证码最多允许 5 次错误尝试，超过后该验证码作废，需重新发送。

---

### 3.3 邀请码管理

#### 3.3.1 批量生成邀请码

**用户故事**：作为品牌管理员，我希望批量生成一组邀请码，并可为受邀用户预设会员等级。

**业务规则**：

- 支持一次生成 1–500 个邀请码。
- 生成时可选设置：
  - 过期时间
  - 渠道标签（如"公众号推广"、"内测"等）
  - **预设会员等级**（free / monthly / yearly / lifetime），默认 free
  - 预设会员等级为 monthly 或 yearly 时，需指定会员时长（如 30 天、365 天）
  - 预设会员等级为 lifetime 时，无需指定时长
- 邀请码格式为短字符串（如 `XK7F-9M2P`），易于传播。
- 每个邀请码关联生成者（管理员）信息。

#### 3.3.2 查看邀请码状态

**用户故事**：作为品牌管理员，我希望查看所有邀请码的使用情况，了解增长数据。

**展示信息**：

- 邀请码、创建时间、过期时间
- 预设会员等级
- 状态（未使用 / 已使用 / 已过期）
- 使用者手机号、使用时间
- 渠道标签
- 支持按状态、渠道、预设等级筛选

### 3.4 会员体系

#### 3.4.1 会员等级定义

| 等级 | 标识 | 说明 |
|------|------|------|
| 免费 | `free` | 注册即获得（邀请码未指定等级时），可使用子产品基础功能 |
| 月度会员 | `monthly` | 按月订阅或邀请码授予，解锁全部功能 |
| 年度会员 | `yearly` | 按年订阅或邀请码授予，解锁全部功能 |
| 终身会员 | `lifetime` | 仅通过邀请码或管理员手动授予，永不过期，解锁全部功能 |

#### 3.4.2 订阅与续费

**用户故事**：作为普通用户，我希望订阅会员以解锁子产品的全部功能。

**业务规则**：

- 用户可从 Free 升级到月度或年度会员。
- 终身会员仅可通过邀请码预设或管理员手动授予，不可自助购买。
- 会员资格记录开始时间和到期时间（终身会员无到期时间）。
- 同一用户同一时刻只能有一个有效会员资格。
- 会员到期前可续费，新周期从当前到期时间开始累加。
- 会员状态变更后，在下一次凭证刷新时自动生效于所有子产品。

#### 3.4.3 取消订阅

**业务规则**：

- 取消订阅后，当前周期内会员资格仍然有效，到期后降为 Free。
- 取消操作不可撤销，但可重新订阅。
- 终身会员不可取消。

#### 3.4.4 会员状态传递

**核心机制**：

- 会员等级和到期时间嵌入访问凭证中。
- 子产品通过解析凭证获取会员状态，**无需向品牌中台发起额外查询**。
- 会员状态的更新存在一个等于凭证有效期的最大延迟（可接受）。

---

### 3.5 用户资料管理

#### 3.5.1 查看与编辑个人资料

**用户故事**：作为已注册用户，我希望查看和编辑我的昵称。

**品牌层管理的字段**：

- 昵称（nickname）——同时嵌入 Token，子产品可直接读取
- 手机号（仅查看，暂不支持修改）

**说明**：子产品如有自身特有的用户偏好（如饮食偏好、健身目标），应在子产品内部存储，以 `user_sub` 关联。

---

### 3.6 管理后台

#### 3.6.1 用户管理

**管理员能力**：

- 查看用户列表（支持搜索、分页）
- 查看用户详情（注册时间、会员状态、使用的邀请码）
- 暂停/恢复用户账号
- 手动授予/延长会员资格（含终身会员）

#### 3.6.2 数据看板（P1，后续迭代）

**展示指标**：

- 总注册用户数、活跃会员数（按等级分布）
- 邀请码使用率（按渠道、按预设等级）
- 会员转化率（Free → 付费）
- 会员续费率

## 4. 子产品接入规范

### 4.1 接入模型

品牌中台与子产品的关系遵循以下模型：

```
品牌中台（Brand Hub）
  ├── 拥有：用户身份、会员资格、凭证签发
  ├── 发布：公钥端点（供子产品验证凭证）
  │
  └── 子产品 A（如饮食管理）
  │     ├── 验证凭证、提取 user_sub、nickname 和会员状态
  │     ├── 不存储用户信息
  │     └── 自主管理业务数据（以 user_sub 为外键）
  │
  └── 子产品 B（如健身追踪）
        └── 同上
```

### 4.2 子产品的职责边界

| 维度 | 品牌中台负责 | 子产品负责 |
|------|-------------|-----------|
| 用户注册/登录 | ✅ | ❌ |
| 短信验证码发送 | ✅ | ❌ |
| 会员订阅/取消 | ✅ | ❌ |
| 凭证签发与刷新 | ✅ | ❌ |
| 凭证验证 | 提供公钥 | ✅ 本地验证 |
| 会员状态判断 | 写入凭证 | ✅ 从凭证读取 |
| 用户昵称获取 | 写入凭证 | ✅ 从凭证读取 |
| 业务数据存储 | ❌ | ✅ |
| 用户表 | ✅ | ❌ 禁止创建 |

### 4.3 子产品可获取的用户信息

子产品通过 Token 可获取以下用户信息，无需额外请求品牌中台：

| 字段 | 来源 | 说明 |
|------|------|------|
| `sub` | Token 标准字段 | 用户全局唯一标识（UUID） |
| `nickname` | Token 自定义字段 | 用户昵称，用于展示 |
| `brand.membership` | Token 自定义字段 | 会员等级（free / monthly / yearly / lifetime） |
| `brand.membership_exp` | Token 自定义字段 | 会员到期时间（终身会员为 null） |
| `brand.roles` | Token 自定义字段 | 角色列表 |

### 4.4 接入流程

一个新子产品接入品牌生态需完成以下步骤：

1. **注册产品标识**：向品牌中台注册产品名称（如 `eat-healthy`），该名称将作为凭证的受众（audience）标识。
2. **获取公钥端点**：配置品牌中台的公钥发现地址，用于本地验证用户凭证。
3. **实现凭证验证中间件**：在子产品的 API 网关层实现凭证验证逻辑。
4. **建立业务数据模型**：所有与用户关联的业务表使用 `user_sub`（UUID）作为标识符，不建立物理外键约束。
5. **实现会员权限判断**：根据凭证中的会员等级控制功能访问。

### 4.5 子产品接入的禁止事项

- ❌ 不得创建用户表或存储任何用户身份信息（手机号等）。
- ❌ 不得自行实现注册、登录流程。
- ❌ 不得自行发送短信验证码。
- ❌ 不得直接修改会员状态。
- ❌ 不得将凭证中的敏感信息持久化存储。
- ❌ 不得在子产品之间直接传递用户数据，应通过品牌中台中转。

## 5. 非功能需求

### 5.1 性能

- 登录接口响应时间 ≤ 500ms（P99，不含短信通道延迟）。
- 验证码发送响应时间 ≤ 1s（P99）。
- 凭证验证为本地操作，子产品不因鉴权增加网络延迟。
- 公钥端点支持客户端缓存，减少请求频率。

### 5.2 安全

- 无密码系统，短信验证码为唯一认证因子。
- 验证码具备严格的发送频率限制和错误次数限制。
- 凭证使用非对称加密签名（私钥仅品牌中台持有）。
- 所有 API 通信强制 HTTPS。
- 敏感操作（发送验证码）实施频率限制。
- 管理后台接口独立鉴权，限制访问来源。

### 5.3 可用性

- 品牌中台目标可用性 ≥ 99.9%。
- 品牌中台不可用时，已登录用户仍可正常使用子产品（凭证本地验证）。
- 仅在凭证刷新时依赖品牌中台在线。
- 短信服务商不可用时，应有备用通道或明确提示。

### 5.4 可扩展性

- 支持未来新增子产品无需修改品牌中台核心逻辑。
- 支持未来扩展 OAuth2/OIDC 标准协议（为第三方登录预留）。
- 会员体系支持新增等级和权益定义。

## 6. 数据边界与隐私

### 6.1 数据归属

| 数据类型 | 归属 | 存储位置 |
|----------|------|----------|
| 账号信息（手机号、昵称） | 品牌层 | brand_db |
| 邀请码 | 品牌层 | brand_db |
| 会员资格 | 品牌层 | brand_db |
| 短信验证码 | 品牌层 | brand_db（或 Redis） |
| 业务数据（如饮食计划、健身记录） | 子产品层 | 各子产品独立 DB/Schema |

## 7. 里程碑规划

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| **P0 — MVP** | 手机号+短信验证码注册/登录、邀请码（含预设会员等级）、凭证签发/刷新、会员体系（Free/Monthly/Yearly/Lifetime）、公钥端点、首个子产品接入 | 必须 |
| **P1 — 管理能力** | 管理后台（邀请码管理、用户管理、手动授予会员、数据看板） | 高 |
| **P2 — 体验增强** | 修改手机号、邮箱绑定、会员年度订阅自助购买 | 中 |
| **P3 — 生态扩展** | OAuth2/OIDC 标准协议、第三方登录（微信/Google）、多租户支持 | 低 |
